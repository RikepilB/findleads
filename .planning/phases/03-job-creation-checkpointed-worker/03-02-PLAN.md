## Phase Goal

**As a** solo lead-generation user (Richard), **I want to** submit a category + free-text location
scrape request and have it run as a safe, resumable background job, **so that** I can kick off a
Places API scrape without the request blocking on network calls or a long-running job silently
dying past Vercel's execution-time ceiling.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| The `after()` callback boundary â€” no HTTP response exists to attach an error to once inside it | `runScrapeJob` is this phase's only code running past that boundary; it must never let an error escape uncaught |
| Places API response content (business names/addresses) flowing into `upsertBusiness`/`insertLeadSnapshot` | Already-validated (Phase 2's Zod schema) but still externally-sourced data reaching the DB write path this task calls |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Denial of Service (to the user's own visibility) | An uncaught error inside `runScrapeJob`, invoked from `after()`'s callback with no response to attach an error to, leaving a job silently stuck at `pending`/`running` forever | high | mitigate | The entire loop body is wrapped in one `try/catch` that always resolves to a terminal `status: 'error'` with a reason â€” no code path inside `runScrapeJob` can throw past this boundary (Pitfall 1) |
| T-02-02 | Information Disclosure | `error_reason` storing a raw, unfiltered `err.message` from an unexpected error type (e.g. a Postgres/Neon driver error containing connection details), later rendered directly in a future CRM/job-history UI | medium | mitigate | Only known-safe error types (`PlacesApiError`, Zod errors) pass their `.message` through as-is; every other error is replaced with a fixed generic string (`'Unexpected worker error'`) before being persisted (Pitfall 2) |
