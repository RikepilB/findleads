## Phase Goal

**As a** solo lead-generation user (Richard), **I want to** submit a category + free-text location
scrape request and have it run as a safe, resumable background job, **so that** I can kick off a
Places API scrape without the request blocking on network calls or a long-running job silently
dying past Vercel's execution-time ceiling.

<objective>
Wire the `POST /api/jobs` Route Handler (SCRAPE-01, JOB-01) on top of Plan 03-02's `runScrapeJob`,
and prove the full pipeline — route to DB to checkpointed worker to Places fixtures to
leads/businesses rows — end-to-end against a real test database.

Purpose: this is the plan that completes Phase 3's vertical slice — by the end of this plan,
submitting a category+location request via `POST /api/jobs` produces a real, checkpointed,
resumable scrape job, not just unit-tested internals.
Output: `app/api/jobs/route.ts` and `tests/integration/jobs/runScrapeJob.test.ts` — both passing,
the latter exercising a real Neon test database.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `POST /api/jobs`'s request body (untrusted, unauthenticated caller) -> `category`/`location` | This phase's one new untrusted-input boundary — the request body is attacker- or at least caller-controlled |
| The billed Google Places API, reachable transitively via any accepted `category`/`location` | Unbounded input here has a real dollar cost, not just a correctness concern |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Denial of Service / Tampering | Unbounded or malicious `category`/`location` strings sent to Google's billed Text Search API via `textQuery` (cost amplification) | high | mitigate | `createJobSchema` (`z.string().trim().min(1).max(200)`) bounds both fields in `POST /api/jobs` before they ever reach `buildTextQuery`/`searchTextPlaces` |
| T-03-02 | Information Disclosure | The `jobId` returned by this phase's `POST /api/jobs` is only meaningful in combination with a future read endpoint — no `GET /api/jobs/:id` exists in this phase's scope, so ID enumeration isn't yet exercisable | low | accept | `jobs.id` is already `uuid().defaultRandom()` (Phase 1 schema decision, carried) — no additional action needed here; Phase 4's `GET` endpoint is the actual future enforcement point for this threat |
