import 'server-only'

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `.env`/`.env.test` (local filesystem) -> server-side process | Secrets cross from disk into the Node process; must never cross further into any client bundle |
| Leaked/overly-broad API key -> Google Cloud billing/quota | If `PLACES_API_KEY` leaks, its blast radius is bounded (or not) by the Cloud Console restriction configured in this plan |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-01 | Information Disclosure | Secret leakage into the client JS bundle | high | mitigate | `lib/env.ts` is the sole `process.env` reader (enforced by convention + grep check), `import 'server-only'` blocks accidental client-side import at build time, neither var name uses the client-inlining prefix convention |
| T-02-02 | Elevation of Privilege | Overly-broad blast radius if `PLACES_API_KEY` leaks (committed, logged, or exposed via a misconfigured route in a later phase) | high | mitigate | SEC-02 Cloud Console restriction to Places API (New) only, verified with a live positive+negative API call pair in Task 3, not just a manual "I clicked save" claim |
| T-02-03 | Information Disclosure | `DATABASE_URL`/`TEST_DATABASE_URL` exposure via accidental commit | medium | mitigate | Both live only in gitignored `.env`/`.env.test`; `.env.example` carries empty placeholders only; Task 1's verify step confirms `git check-ignore` succeeds on both real secret files |
</threat_model>
