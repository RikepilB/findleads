<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Untrusted Places API content (future phases) -> `businesses`/`leads` columns defined here | This phase defines the columns that will hold untrusted, externally-sourced business names/addresses starting Phase 2 |
| Unauthenticated `GET /api/jobs/:id` (Phase 3+) -> `jobs.id` in the URL | The job ID is exposed directly in a polled URL with no auth layer, per PROJECT.md's locked no-auth decision |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-01 | Tampering | SQL injection via untrusted Places API content (business names/addresses) eventually landing in `businesses`/`leads` columns | high | mitigate | Drizzle's query builder parameterizes all values by construction; the schema/DAL functions (this plan and Plan 01-04) use the parameterized `.values()`/`.set()` API exclusively ã `sql.raw()` on untrusted input is never used anywhere in this phase |
| T-03-02 | Information Disclosure | IDOR via sequential/guessable job IDs on the future unauthenticated `GET /api/jobs/:id` polling endpoint | medium | mitigate | `jobs.id` is `uuid().defaultRandom()`, decided in this plan's schema even though the route itself ships in Phase 3 ã enumeration of other jobs' data is not feasible from a sequential ID |
</threat_model>
