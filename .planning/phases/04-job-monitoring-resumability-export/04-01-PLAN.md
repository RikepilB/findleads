## Phase Goal

**As a** solo lead-generation user (Richard), **I want to** watch a scrape job through to
completion — with automatic resumption of a partial job, honest zero-result/failure states, and a
CSV of finished leads — **so that** I never have to babysit a stuck job or guess whether "no
results" means the scrape worked or silently broke, and can pull the leads out to start outreach.

<objective>
Build the atomic-claim continuation trigger (JOB-04), the stale-job watchdog (JOB-05), and the
zero-result-vs-error response contract (JOB-06) on top of Phase 3's already-shipped, already-
verified `runScrapeJob`/`getJob`/`updateJobProgress`. This is the first `GET /api/jobs/:id` Route
Handler in the codebase — the endpoint Richard's future poller (Phase 5) hits every ~1s to watch a
job to completion.

Purpose: Phase 3 made a scrape safe and resumable; this plan makes that resumption actually
trigger from client traffic (no cron/queue), and makes a stuck or zero-result job legible instead
of ambiguous.
Output: `lib/jobs/checkpoint.ts`'s `WATCHDOG_MS`, `lib/db/jobs.ts`'s `claimPartialJob`/
`flagStaleJob`, and `app/api/jobs/[id]/route.ts`'s `GET` handler — all typechecking, unit- and
integration-tested against a real Neon test database.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `jobs.id` path param (untrusted, unauthenticated caller) -> `eq(jobs.id, id)` | Flows straight into a Drizzle parameterized query — no raw SQL interpolation; `uuid` type mitigates trivial enumeration (carried from Phase 1/3) |
| Two concurrent `GET` requests -> `jobs` row's `status` column | Race surface for double-scheduling a billed Places API continuation |
| `jobs.cursor` (internal, Google-issued `pageToken`) -> public JSON response | Internal resumption detail that must never reach the response body |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-04-01 | Tampering / DoS (cost amplification) | Two concurrent `GET /api/jobs/:id` polls both scheduling `runScrapeJob` for the same `partial` job, burning Places API quota | high | mitigate | `claimPartialJob`'s conditional `UPDATE ... WHERE status='partial' RETURNING *` guarantees only one poll ever wins the race and schedules `after()` (Pattern 1, Task 1/2) |
| T-04-02 | Denial of Service (to the user's own visibility) | A crashed/orphaned `after()` invocation leaves a job silently `pending`/`running` forever, with no error surfaced anywhere | medium | mitigate | `flagStaleJob`'s conditional `UPDATE ... WHERE updated_at < now() - WATCHDOG_MS RETURNING *`, triggered on every `GET`, flips it to `error` with a human-readable reason (Pattern 2, Task 1/2) |
| T-04-03 | Information Disclosure | `jobs.cursor` (a raw, opaque Google-issued `pageToken`) leaking into the public JSON response via a `SELECT *`-style spread | low | mitigate | `GET` handler explicitly shapes the response object field-by-field, never spreads the raw row (Task 2, Pitfall 6) |
| T-04-04 | Spoofing / Access Control | `jobs.id` enumeration by an unauthenticated caller | low | accept | `jobs.id` is `uuid().defaultRandom()` (Phase 1 schema decision, carried) — no auth exists or is planned for this single-user v1 tool |
</threat_model>
