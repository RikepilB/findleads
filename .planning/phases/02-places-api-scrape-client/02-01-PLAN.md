<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Google Places API response → `lib/places/schema.ts` | Untrusted external JSON parsed and validated before any downstream code (Plan 02-03/02-04) ever reads a field |
| Test environment (Vitest) → `lib/env.ts` import | Test-time environment variable shape must satisfy `lib/env.ts`'s Zod schema or the module throws at import, breaking every test that transitively imports it |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-------------------|
| T-02-01-01 | Tampering | `lib/places/schema.ts` (`rawPlaceSchema`, `textSearchResponseSchema`) | medium | mitigate | Zod validation rejects malformed/renamed/wrong-typed fields immediately at parse time rather than letting `undefined`/wrong-typed values propagate silently into `client.ts` (02-03) or `mapPlaceToLead.ts` (02-04) |
| T-02-01-02 | Tampering | `businessStatusSchema` (`.optional()`, not `.default(...)`) | medium | mitigate | Encodes Pitfall 3 correctly at the schema layer — an absent `businessStatus` stays `undefined`, never silently coerced to `'OPERATIONAL'` or any closed value, which would otherwise cause over- or under-exclusion in `mapPlaceToLead.ts` |
| T-02-01-03 | Denial of Service | `textSearchResponseSchema.places` (unbounded array) | low | accept | No array-length cap is enforced here; Google's own API already caps at 20 results/page and 60/query, and the Places API is a trusted first-party Google endpoint, not arbitrary user-controlled input — an oversized response is not an attacker-controlled vector in this integration |
| T-02-01-SC | Tampering | npm/pip/cargo installs | n/a | n/a | No new packages are installed in this plan — `zod`, `server-only`, `vitest` are already present from Phase 1; no package-legitimacy checkpoint is required here |
</threat_model>
