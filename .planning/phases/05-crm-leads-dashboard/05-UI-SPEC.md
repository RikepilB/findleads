## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label/meta | 12px | 400 regular | 1.5 |
| Body | 14px | 400 regular | 1.5 |
| Heading (section/card title) | 20px | 600 semibold | 1.2 |
| Display (page title, e.g. "Leads") | 28px | 600 semibold | 1.2 |

Exactly 2 weights used across the whole phase: 400 (regular, all body/label/data text) and 600
(semibold, all headings, badges, and button labels). Body text is 14px, not 16px — this is a
dense, data-heavy internal table tool (leads list, job history), and 14px keeps more rows
visible without scrolling, matching the "functional, not a polished SaaS" framing in
PROJECT.md/CLAUDE.md.

## Attribution Requirement (SEC-03)

Per Google Maps Platform's attribution requirements (confirmed against the official
`developers.google.com/maps/documentation/places/web-service/policies` page during Phase 5
domain research — corrects an earlier draft of this spec that used non-standard copy),
any UI displaying Places API-sourced content must show Google attribution, even with no map
rendered. Contract:

- Text: **"Google Maps"** (the official required attribution string — NOT "Powered by
  Google"), 12px, gray-500, un-styled as a link (no need to link out for v1 — plain text
  attribution satisfies the requirement when no interactive map is shown).
- Placement: adjacent to each table displaying Places-sourced content (Leads table and Job
  History table), not solely in a distant global footer — a single global placement risks
  reading as detached from the specific content it's attributing. A persistent app-layout
  placement (e.g. a slim bar under the top nav, visible on every page) satisfies this as long
  as it's visually associated with the content area, not buried at the bottom of a long page.
- Do not modify, recolor, or shrink below 12px. Do not place behind a collapsed menu.

## Page/View Inventory (informational — not part of the strict template, included because this
is the first UI phase and no prior page structure exists)

- **Leads** (`/` or `/leads`) — default view. Table: business name, tier badge, phone, address,
  rating, contacted toggle, notes (inline-editable), `updated_at` (relative, e.g. "2h ago", to
  show staleness per CRM-04).
- **Job History** (`/jobs`) — job-creation form (category + location + "Start Scrape") above a
  table: status badge, category, location, `leads_found`, `created_at`, SCRAPE-07 cap notice
  when applicable, "Export CSV" link when `done`.
- Simple two-tab top nav ("Leads" / "Job History") — no sidebar; single user, two views, no
  need for nested navigation.
- Google attribution footer present in the shared layout, applies to both views.

## Assumptions & Flags

Explicit calls where upstream artifacts didn't fully specify — flagged per instructions rather
than left as blocking placeholders:

1. **shadcn not initialized.** No `components.json` exists and this is a fully autonomous
   session (no user turn available to paste a shadcn preset string). Given the "keep it simple,
   personal internal tool" framing and a small, table-heavy UI surface, this spec is written
   against plain Tailwind utilities so the executor is never blocked waiting on an interactive
   step. If a human wants shadcn's `Table`/`Switch`/`Badge`/`Tabs` primitives before
   implementation, that's a cheap upgrade — re-run the shadcn gate manually first.
2. **A job-creation form is in scope for this phase, even though no requirement names it.**
   STATE.md's own Blockers/Concerns section flags that SCRAPE-01 is satisfied at the API layer
   only (`POST /api/jobs`) and no job-creation UI requirement is tracked. Without *some* form,
   Richard has no way to actually use the product end-to-end (the whole point per PROJECT.md's
   Core Value). This spec includes a minimal category+location form on the Job History view as
   necessary, incidental scope — flagged here so planning doesn't mistake it for a tracked
   requirement it can defer.
3. **SCRAPE-07 cap-detection trigger is a derived heuristic, not an existing flag.**
   Confirmed in `lib/jobs/checkpoint.ts` (`MAX_PAGES = 3`) and `lib/jobs/runScrapeJob.ts`: the
   worker stops paginating at 3 pages (Google Text Search returns up to 20 results/page = 60
   max), and the job's `cursor` is cleared to `null` once `status` becomes `done` — there is no
   stored boolean distinguishing "hit the 60 cap" from "ran out of results naturally." Recommend
   the UI/API layer trigger the SCRAPE-07 message when `leads_found >= 60 && status === 'done'`.
   Flagging for the planner/executor: this is a reasonable proxy given the fixed page size, but
   confirm against the actual last-page result count if a false positive (exactly 60 organic,
   no more available) matters — the requirement's own wording ("refine your search") already
   reads as advisory, not a hard guarantee, so the heuristic is acceptable as-is.

   **Superseded (2026-07-03):** 05-RESEARCH.md's domain research found this heuristic has a
   false-negative mode (closed-business filtering can drop `leads_found` below 60 even when
   the raw cap was hit) and replaced it with an additive `jobs.resultCapHit` boolean computed
   from the raw pagination signal before filtering — see 05-01-PLAN.md Task 2. 05-04-PLAN.md
   renders the cap message off `resultCapHit`, not this heuristic.
4. **No destructive actions exist in this phase's scope.** CRM-02/CRM-03 (notes, contacted
   toggle) are reversible, non-destructive edits — no confirmation dialog needed. There is no
   delete-lead, delete-job, or delete-note action in v1. If one is added later, it needs its own
   destructive-color + confirmation-copy entry; not needed now.
5. **Notes autosave vs. explicit Save button:** chose autosave-on-blur (see Copywriting
   Contract) over an explicit "Save" button to minimize clicks in a personal tool used for rapid
   outreach triage. If this proves error-prone in practice (e.g., accidental blur losing intent
   clarity), an explicit button is a trivial swap — not a structural change to the contract.
