<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser form submit -> Server Action | User-controlled `businessId`/`notes`/`contacted` form fields cross into server code here |
| Places-sourced business name/address rendered in JSX | Untrusted upstream string content (already persisted by Phase 1-3) rendered directly in the table |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-05-05 | Tampering | updateNotesAction / setContactedAction | high | mitigate | zod schemas (`businessId` coerced positive int, `notes` trimmed+capped at 2000 chars, `contacted` coerced boolean) reject malformed input before any DAL call (Task 1) |
| T-05-06 | Spoofing | Server Action invocation (CSRF) | medium | accept | Next.js Server Actions perform automatic origin-header verification since 13.4+ (research Assumption A1); no additional manual CSRF token added for this internal single-user tool |
| T-05-07 | Tampering / Information Disclosure | XSS via untrusted business name/address/notes rendered in the table | high | mitigate | React's default JSX text-node rendering auto-escapes all string content; no `dangerouslySetInnerHTML` used anywhere in this page (verify at code review) |
| T-05-08 | Tampering | businessId in hidden form field can target any business (no per-user ownership) | low | accept | No auth boundary exists in this single-user app by design (PROJECT.md); worst case is editing another business's notes, a UX-integrity concern not a security boundary — DAL still validates `id` is a positive integer |
</threat_model>
